1. Introduction

1.1 Purpose
The purpose of this Software Requirements Specification (SRS) document is to formally
define the functional and non-functional requirements for the Smart University Bus
Information System (SUBIS). This document provides a complete description of the system‚Äôs
features, operational behavior, constraints, and interactions with users and external
components.‚Äã
 It serves as a contract between the development team and the stakeholders, including the
project supervisor, students, transport administrators, and system designers.

The intended audience of this SRS includes:

   ‚óè‚Äã Project Supervisor‚Äã

   ‚óè‚Äã Development Team‚Äã

   ‚óè‚Äã Quality Assurance / Testers‚Äã

   ‚óè‚Äã University Transport Department‚Äã

   ‚óè‚Äã Future maintainers or researchers‚Äã




1.2 Scope
The Smart University Bus Information System (SUBIS) is a real-time, GPS-enabled
transportation information system designed specifically for university students and transport
administrators.‚Äã
 The system provides dynamic bus tracking, estimated time of arrival (ETA), real-time bus
routes, stop information, and crowdedness indicators. The system also leverages
graph-based routing algorithms, mapping/localization technologies, and an LLM-based
interface for user support.

The main objectives of SUBIS are:

   ‚óè‚Äã To provide students with accurate, up-to-date bus locations and ETAs.‚Äã

   ‚óè‚Äã To improve transportation efficiency by analyzing route patterns, crowd levels, and
      flow.‚Äã
   ‚óè‚Äã To reduce waiting time and uncertainty for students at bus stops.‚Äã

   ‚óè‚Äã To provide transport administrators with an analytical dashboard to monitor routes,
      buses, and performance.‚Äã

   ‚óè‚Äã To create an end-to-end system combining, backend services, and AI-powered
      interaction(optional).‚Äã


SUBIS will not perform bus scheduling, driver management, or fare/payment processing. Its
sole purpose is information collection, processing, and delivery.

2. Overall Description

2.1 Product Perspective
The Smart University Bus Information System (SUBIS) is an independent software product
designed to improve and optimize the transportation information system within the university
environment. SUBIS replaces manual timetable checking, unreliable student-to-student
communication, and traditional static bus schedules with a dynamic, real-time, intelligent
solution.

The system integrates multiple components:
   ‚óè‚Äã A web application for students.‚Äã

   ‚óè‚Äã A backend server for processing, storing, and distributing information.‚Äã

   ‚óè‚Äã A university admin dashboard for route management and analytics.‚Äã

   ‚óè‚Äã An LLM-based assistant for natural language queries (optional but included in
      scope).‚Äã

   ‚óè‚Äã Real-time GPS modules or mobile-based location tracking for students and
      buses.‚Äã


SUBIS communicates with external systems such as:

   ‚óè‚Äã OpenStreetMap (OSM) tile servers for map rendering.‚Äã

   ‚óè‚Äã University email services for authentication.‚Äã

   ‚óè‚Äã WebSocket (Socket.io) servers for real-time updates.‚Äã


The system is independent, modular, and does not require integration with existing
university ERP systems.




2.1.1 Operations

SUBIS will operate in the following modes:

   ‚óè‚Äã Student Interactive Mode‚Äã
       Students open the web app to view bus locations, ETAs, stop information, and route
      suggestions. They may also enable GPS sharing for improved crowd prediction.‚Äã

   ‚óè‚Äã Real-Time Monitoring Mode‚Äã
       The backend continuously collects GPS data from students and buses, processes
      routes, updates ETAs, and pushes live updates to all connected devices.‚Äã

   ‚óè‚Äã Admin Operational Mode‚Äã
      Transport administrators can add/edit routes, analyze usage, check bus density, and
      monitor system health via the admin dashboard.‚Äã

   ‚óè‚Äã Unattended Operations‚Äã
      Even when no users are active, the backend continues logging telemetry, running
      analytics, and updating route statuses.‚Äã

   ‚óè‚Äã Backup & Recovery‚Äã
      Database snapshots and logs will be taken regularly. In case of downtime, the
       system will recover from the latest saved state.‚Äã




2.1.2 Site Adaptation Requirements

To deploy SUBIS, the following site-specific adaptations may be required:

   ‚óè‚Äã Availability of a university-hosted server or cloud environment (e.g., VPS, local
      machine, AWS EC2).‚Äã

   ‚óè‚Äã Pre-loading official university bus routes and stop coordinates into the database.‚Äã

   ‚óè‚Äã GPS-enabled devices for students.‚Äã

   ‚óè‚Äã Stable Wi-Fi or mobile data coverage across the campus.‚Äã


For Location HTML GPS API will be used.




2.2 Product Functions
SUBIS provides the following major system functions:

   ‚óè‚Äã Real-Time Bus Tracking‚Äã
      Display current bus locations on an interactive map.‚Äã

   ‚óè‚Äã ETA Calculation‚Äã
       Estimate arrival time for a selected stop based on bus speed, distance, and route
      topology.‚Äã

   ‚óè‚Äã Route Visualization & Path Finding‚Äã
      Display available bus routes, stops, and required transfers if direct routes are
      unavailable.‚Äã

   ‚óè‚Äã Student GPS Location Collection (Optional)‚Äã
      Improve crowd estimation and route prediction by collecting user locations (opt-in).‚Äã

   ‚óè‚Äã Crowd Level Estimation‚Äã
      Show whether a bus or stop is crowded, moderate, or free using aggregated user
      data.‚Äã

   ‚óè‚Äã Admin Dashboard‚Äã
      Manage routes, buses, schedules, and view analytics (usage, peak times, crowd
       heatmaps).‚Äã

   ‚óè‚Äã Notifications System‚Äã
      Send alerts for approaching buses, delays, and route changes.‚Äã

   ‚óè‚Äã LLM-based Query Assistance‚Äã
       Students can ask questions like ‚ÄúWhen will Bus 12 reach Gate #1?‚Äù or ‚ÄúWhich bus
      should I take to Department X?‚Äù‚Äã


These functions collectively provide an intelligent transportation information experience.

2.4 General Constraints
The system is constrained by the following factors:

   ‚óè‚Äã Regulatory / Institutional Policies‚Äã
      GPS data must be collected only with consent and handled according to privacy
      guidelines.‚Äã

   ‚óè‚Äã Hardware Limitations‚Äã
      Students‚Äô devices may have low GPS accuracy; system must tolerate location noise.‚Äã

   ‚óè‚Äã Internet Connectivity‚Äã
       University areas with weak signals may cause temporary real-time disruptions.‚Äã

   ‚óè‚Äã Performance Constraints‚Äã
      Real-time updates must remain within 1‚Äì3 seconds of actual bus movement.‚Äã

   ‚óè‚Äã Mapping Engine Constraint‚Äã
      Must use open-source maps (OSM) to avoid licensing costs.‚Äã

   ‚óè‚Äã Security Constraints‚Äã
      Communication must be encrypted (HTTPS, WebSocket Secure).‚Äã

   ‚óè‚Äã Resource Constraints‚Äã
      University infrastructure may limit server hosting options.‚Äã

   ‚óè‚Äã Internet connectivity is sufficient for real-time communication.‚Äã
